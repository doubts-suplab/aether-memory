Memory Model

Four Types of Shared Memory

The four-type cognitive model of Aether Core, generalised from an individual to a group. Every memory is embedded (384-dim), stored in PostgreSQL with pgvector, strengthened by collective access, and decayed when the team stops relying on it.

Episodic

Team Events

Shared experiences: "the Q3 incident retrospective and what we changed after it."

Semantic

Org Facts

Organisational knowledge: "our SLA is 99.95% and the escalation path is on-call → lead."

Procedural

Playbooks

Shared conventions: "how the team ships — trunk-based, gated by the quality workflow."

Emotional

Team Affect

Collective sentiment: "morale after the launch — energised, some burnout risk flagged."

Access Control

Visibility Widens Outward — Sharing Is Opt-In

A memory's reach is explicit. Federation only ever sees the outermost ring, and only for tenants that have opted in — privacy is the default.

PRIVATE

Visible only to the owning team (tenantId + teamId).

→
TENANT

Visible to every team within the same tenant.

→
FEDERATED

Eligible for privacy-preserving cross-instance queries — bounded summaries, coarse provenance, no team identity.

Architecture

Four Maven Modules

Strict inward dependency: memory-api → memory-engine → memory-domain. The domain is pure Java 21 — no framework coupling.

memory-domain

Domain & Ports

SharedMemory, MemoryScope, MemoryVisibility, MemoryPolicy, FederatedMemory + port interfaces. No Spring.

memory-engine

Adapters & Services

pgvector store (reinforce-on-read), Ollama embedding, policy-aware decay/archive, privacy-preserving federation.

memory-api

REST API · Port 8083

Spring Boot app, controllers, Flyway migrations, scheduled lifecycle, Micrometer metrics.

memory-infra

Infrastructure

Docker Compose, Kubernetes manifests, standalone migration copies. No Java sources.
